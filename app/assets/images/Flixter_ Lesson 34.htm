Lesson 34: Adding a Teacher Dashboard

We already have built the core of our video streaming platform. Users on our platform are able to create, buy, and watch user uploaded video lessons.

Now that the core features of our web application are built and we have a working application we can start focusing on aspects of our User Experience that we can improve.

In this lesson, we will add a dashboard where both students and instructors can see an overview of the courses they're teaching and the courses they are enrolled in.

Whenever we build a new page in our web application, it's a good idea to look at the wireframes of our application and to take a moment to think about what exactly we want to build. These are the wireframes for the page we are going to implement in this lesson:

Teacher Dashboard


Generating the Dashboards Controller

Whenever you want to build a new page on your web application, the first question you should ask yourself is what controller will the page live in and what action will it use. In this case we should make a new controller for our dashboard called the dashboards controller. Run the following command to generate it: 

$ rails generate controller dashboards

When the command completes a new file for the dashboards controller will be generated. After the controller is created we will usually hook up the routes file. In this case, we're going to throw a curveball at you and hook up the routes file a little differently than what we have in the past...

Singleton Routes

Thinking through the page that we are building, it would make sense to have this page be on the show action of the dashboards controller. This show action is different than other show actions we've built in the past. Rather than passing through the id of the user in the URL, something like /dashboards/1, which is what we typically would do, we can show the dashboard for the current_user. This means we don't need to pass the id of the user in the URL, because we don't need to use it. This means that the URL that looks like /dashboard would be better suited for us than passing the id through.

Setting the routes for the show action that do not pass through an id happens less frequently than the typical case, but still enough to warrant a discussion. Ruby on Rails supports this case using what are known as singleton routes. We're going to setup the dashboards controller's action as a singleton route. As you see the example in action it will become clearer how this works.

Edit config/routes.rb to add support for the show action in the dashboards controller: 

Rails.application.routes.draw do
  devise_for :users
  resource :dashboard, only: [:show]
  root 'static_pages#index'
  resources :courses, only: [:index, :show] do
    resources :enrollments, only: :create
  end
  resources :lessons, only: [:show]
  namespace :instructor do
    resources :lessons, only: [:update]
    resources :sections, only: [:update] do
      resources :lessons, only: [:new, :create]
    end
    resources :courses, only: [:new, :create, :show] do
      resources :sections, only: [:new, :create]
    end
  end
end

Save the file.

You will notice two key differences on this line, as opposed to the line we usually would write in the file:

  • Rather than using the command resources, the command resource is used.

  • When we specify the name of the controller we're linking to we specify the singular version of the word, even though the controller name we generated is plural. In this case it will connect to our dashboards controller, even though we specified :dashboard in the routes, which is counterintuitive.

Whenever we add lines to the routes file, we expect the output of rake routes to change. Run the command to output the routes table: 

$ rake routes

Study the output of the routes table and find the new line that our change added. It looks like this: 

dashboard GET    /dashboard(.:format)                                   dashboards#show

The line we added connected requests to /dashboard on our web application to go to the dashboards controller's show action and does not require an id to be passed along as params in the URL.

Singleton routes interact with the other typical RESTful URLs in addition to show: new, create, edit, update and destroy. If you want to dig a little deeper into singleton routes, read Chapter 3.6 of the Rails 3 Way.

Fleshing out the Dashboard Controllers and Views

Add an empty show action to the dashboards controller. Edit app/controllers/dashboards_controller.rb to look like this: 

class DashboardsController < ApplicationController
  def show
  end
end

Save the file.

Add a view for the show action so we have a place to put the HTML for the user dashboard. Add the following HTML and save the new file as app/views/dashboards/show.html.erb: 

<br />

<div class="booyah-box col-10 offset-1">
  <h1>My Dashboard</h1>
</div>

Save the file. If you manually type localhost:3030/dashboard you should see the new page load.

Rather than requiring users to manually type in the URL, we should provide the user access to this page as a link that lives in the top navigation. The first step of building a link is to determine the destination of the link. To find the destination of the link, look up the value in the routes table. Run the following command to output the contents of the routes table. 

$ rake routes

We want to send the user to their dashboard, or in other words the dashboards controller and the show action. The line in the routes table that represents that looks like this: 

dashboard GET    /dashboard(.:format)                                   dashboards#show

This means the link should go to the dashboard_path. Edit app/views/layouts/application.html.erb so there is a link to the dashboard from the top navigation: 

<ul class="navbar-nav">
  <% if user_signed_in? %>
    <li class="nav-item">
      <%= link_to 'My Dashboard', dashboard_path, class: 'nav-link' %>
    </li>
    <li class="nav-item">
      <%= link_to 'Sign out', destroy_user_session_path, method: :delete, class: 'nav-link' %>
    </li>
  <% else %>
    <li class="nav-item">
      <%= link_to 'Sign in', new_user_session_path, class: 'nav-link' %>
    </li>
    <li class="nav-item">
      <%= link_to 'Sign up!', new_user_registration_path, class: 'nav-link' %>
    </li>
  <% end %>
</ul>

Save the file and refresh the page.

If you click the brand in the top left corner you will navigate to the root page of the site. If you're logged in and you press the My Dashboard link you will reach the dashboard page we just started building.

The dashboard page will show the details of the courses a user (student) is enrolled in and also the courses a user (instructor) is teaching. This information only makes sense if we have a user logged in. Prevent unauthenticated users from accessing the page by adding devise's before_action to app/controllers/dashboards_controller.rb like this: 

class DashboardsController < ApplicationController
  before_action :authenticate_user!

  def show
  end
end

Save the file.

Adding a User's Enrollments to Their Dashboard Page

On the content of the dashboard page we should list out each of the courses the current_user is enrolled in. Because we configured our user model to Have Many enrolled_courses Through..., we can access each of the courses the current_user is enrolled in by writing code that looks like: current_user.enrolled_courses.

Loop through each of the current_user's enrolled courses and add the course details by editing app/views/dashboards/show.html.erb to do this: 

<br />

<div class="booyah-box col-10 offset-1">
  <h1>What I'm Learning</h1>
  <% current_user.enrolled_courses.each do |course| %>
    <h3 class="text-center"><%= course.title %></h3>
  <% end %>
</div>

Save the file and refresh the page. If you're not currently enrolled in a course as the user that is logged in, you will want to enroll in one, by finding a course on your application and pressing the Enroll button. Notice how the course details are shown on the page.

On the user's dashboard make the title of the course link the user to the course detail page. To find the destination of the link, look up the value in the routes table. Run the following command to output the contents of the routes table. 

$ rake routes

We want to send the user to the course detail page, or in other words the courses controller and the show action. The line in the routes table that represents that looks like this: 

course GET    /courses/:id(.:format)                                 courses#show

Edit app/views/dashboards/show.html.erb to include a link to the course_path. 

<br />

<div class="booyah-box col-10 offset-1">
  <h1>What I'm Learning</h1>
  <% current_user.enrolled_courses.each do |course| %>
    <h3 class="text-center">
      <%= link_to course.title, course_path(course) %>
    </h3>
  <% end %>
</div>

Save the file and refresh the page. Notice how the title of the course that you are enrolled in is now a link to the detail page for the course.
